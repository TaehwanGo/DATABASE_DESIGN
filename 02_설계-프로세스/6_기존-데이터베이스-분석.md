## 기존 데이터베이스 알기

- 정보에 대한 조직의 현재 요구사항을 데이터베이스가 지원할지를 결정해야 한다
- 구조적으로 부족한 부분이 있는지를 확인해야 한다
- 데이터베이스가 조직에서 앞으로 발생할 수 있는 정보 요건을 지원할 수 있도록 어떻게 개선할 필요가 있는지를 결정한다

- 기존 데이터베이스를 기반으로 새로운 데이터베이스를 만들 때 고려해야될 사항들
  - 어떤 유형의 데이터 조직에서 사용하는가?
  - 조직에서 데이터를 어떻게 사용하는가?
  - 조직은 어떻게 데이터를 유지하고 관리하는가?
- 위 질문들은 데이터베이스를 설계할 때 사용할 수 있는 핵심적인 정보를 제공해준다
- 기존 데이터베이스를 분석할 때 명심해야할 규칙
  - 새로운 데이터베이스 구조에 대한 근거로 기존 데이터베이스 구조를 적용하지 말 것
    - 기존 데이터베이스와 새로운 데이터베이스 구조는 완전히 달라질 수 있다 !
    - 설계에 들인 노력을 기반으로 구조를 만들자
    - 기존 데이터베이스 구조 내에 숨어있는 모든 문제가 새로운 데이터베이스로 옮겨지기 때문이다
      - 불편한 테이블 구조, 잘못 정의된 관계, 일관성이 없는 필드 명세 등

## 분석 수행

- 분석 프로세스 세 단계
  - 1. 데이터가 수집된 방법을 검토
  - 2. 정보가 표현되는 방법을 검토
  - 3. 사용자 및 관리자와 인터뷰를 진행하는 것
  - 1, 2 단계를 진행할 때에도 다양한 사람들과 이야기를 나눌 필요가 있다
  - 더 깊은 수준의 질문을 하는 시간 : 인터뷰

### 1. 데이터를 수집하는 방법 살펴보기

- 각 문서의 견본을 수집한다
  - 종이 기반으로 데이터를 수집하는 것
  - 웹에서 데이터를 입력하는 것
  - (엑셀)스프레드 시트
- 데이터를 수집할 때 사용하는 모든 컴퓨터 프로그램들을 검토한다
  - (엑셀)스프레드 시트

### 2. 정보가 표현돠는 방식 살펴보기

- 조직이 데이터를 정보로 `보여주는 방법`에 대한 검토
  - e.g.) 스프레드 시트
- 일반적인 정보 출력 방식 세 가지
  - 1. 보고서 : 엑셀, 손
  - 2. PPT
  - 3. 웹 페이지
- 각 화면들에 대하여 별도로 절차를 수행한다
  - 둘 이상의 화면을 합해서는 실수가 발생될 가능성이 있다

### 3. 인터뷰 진행

- 인터뷰가 분석단계에서 중요한 이유
  - 1. 인터뷰는 이전 검토 단계에서 수집한 견본에 대한 자세한 정보를 제공한닫
  - 2. 인터뷰 참가자들은 조직이 데이터를 사용하는 방법에 대한 정보를 제공한다
  - 3. 인터뷰 내용들은 필드와 테이블 구조를 정의할 때 중요한 역할을 한다
  - 4. 미래의 정보 요건을 정의할 때 도움을 줄 것이다

## 사용자와의 인터뷰

사용자 인터뷰는 다음의 네 가지에 초점을 맞추어 진행한다

1. 사용자가 현재 사용하고 있는 데이터의 종류
2. 사용자들이 현재 데이터를 사용하는 방법
3. 분석의 처음 두 단계에서 모은 견본 데이터의 수집
4. 사용자의 일일 작업 시 필요한 정보의 종류

### 데이터 유형과 사용법 검토

- 인터뷰 목적

  - 사용자가 사용하는 `데이터 유형` 확인
  - 작업 수행 시 `데이터를 사용하는 방법` 확인

- 사용자 데이터에 대한 질문을 조금 더 쉽게하기 위해

  - `데이터 집합`과 데이터를 보여주는 `화면 견본`을 사용한다
    - 아직 견본에 대한 이야기를 할 때는 아니다
    - 데이터 집합과 화면 견본은 별도로 다뤄야 한다

- 대화가 진행되는 동안

  - 개방형 질문을 사용하고, 대답에 있는 대상을 식별
  - 각 대상의 특징을 확인하기 위해 구체적인 후속 질문을 사용한다

- 인터뷰를 시작할 때

  - 참석자가 매일 수행하는 작업에 대하여 질문을 한다
  - 참석자가 수행하는 작업에 대하여 전반적으로 설명을 한 후에 그의 작업에 대하여 보다 자세한 설명을 요청한다

- 인터뷰 예시
  - 질문자 : 당신이 매일 수행하는 작업에는 어떤 것들이 있나요?
  - 참석자 : 저는 다양한 *사람들*이 제출하는 토지 이용 *신청서*를 받고, 일지에 작성한 후, 행정관과 공판 일정을 잡아요.
    그리고 상세 신청서와 관련하여 문의가 있다면, 그에 대한 *지원자*를 돕는 일을 하죠
  - 질문자 : 잠시 신청서에 대해서 이야기를 해보죠. 신청서에는 어떤 종류의 내용이 들어가나요?
  - 참석자 : 굉장히 많아요. 신처서의 이름과 종류, 신청서에 해당하는 목적지와 주소, 그리고 위치와 관련된 내용이 있어요
  - 질문자 : 신청서의 종류와 이름과 관련된 내용에 대해서 이야기해주세요
  - 참석자 : 우리가 기록하는 건 네 가지에요. _신청서의 종류_, _세부 분류의 이름_, _프로젝트의 목적_, 그리고 프로젝트의 *상세 내용*입니다

### 견본 검토

- 인터뷰를 완료한 이유 -> 분석 프로세스에서 먼저 수집한 모든 견본들에 초점을 맞춰 이야기를 하면 된다
- 인터뷰를 진행할 때 목적

  - 견본을 통해 객체를 어떻게 표현했는지를 파악
  - 아직 이해하지 못한 견본의 특징을 명확하게 이해
  - 각 견본에 대한 자세한 설명을 듣는 것

- 첫 질문을 하기에 앞서
  - 인터뷰를 시작할 때 정리한 메모를 확인
    - 이야기한 내용이 이제 이야기할 견본과 관련이 있는지를 확인하고 싶다
  - 질문 예시
    - 당신은 앞의 인터뷰에서 모든 고객의 데이터를 이력으로 관리한다고 하셨는데요.
      이 화면에서 앞에서 말한 작업을 수행하는데 어떻게 도움이 되나요?
  - 답변 예시
    - 이 화면은 제가 새로운 고객을 입력하는 화면이에요.
      뿐만 아니라 기존 고객들에 대한 모든 정보를 수정하고 관리하는 화면이기도 합니다
  - 답변의 만족 여부에 따라
    - 만족 => 견본의 상세 설명에 대한 그거로 사용
    - 불만족 => 견본의 쓰임새와 목적을 명확하게 알 때까지 계속해서 관련된 질문을 한다
- 견본에 대한 설명의 예시
  - 이 화면은 모든 고객 데이터를 수집하고 관리하는 데 사용된다
- 개발자는 모든 견본에 대한 설명을 받아야 한다

  - 왜냐하면 설계 프로세스의 이후 단계에서 이것들이 사용될 것이기 때문이다

- 견본에서 이해하지 못한 부분을 절대로 가정하거나 추측하면 안된다
  - 만일 가정이나 추측이 잘못된 것이라면, 소중한 시간을 낭비하고 이후 프로세스 진행에 곤란을 겪을 수 있다
- 견본에 대한 설명을 적는 것은 어려울 수 있다
  - 만약 하나 이상의 대상을 표현한다고 하면, 견본은 복잡하다
